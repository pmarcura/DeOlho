Princípios

Doutrina cívica da interface

Antes de parecer moderno, o DeOlho precisa ser confiável. A interface deve reduzir ambiguidade, não aumentar engajamento por tensão.

Seis regras de experiência

  • Clareza antes de densidade. Volume só entra quando a hierarquia estiver clara.
  • Fonte antes de conclusão. O usuário precisa ver de onde veio a afirmação.
  • Evidência antes de opinião. Interpretação nunca pesa mais que documento.
  • IA sempre identificada. Explicação por IA não parece fato oficial.
  • Sinal não é acusação. Atenção sugere verificação, não culpa.
  • Voltar ao documento. Toda leitura relevante precisa caminho para evidência.

Separação obrigatória

A UI precisa distinguir visual e textualmente:

Fato oficial Explicação por IA Sinal estatístico Notícia Opinião de usuário Revisão manual
Regra de produto Sem fonte, sem afirmação pública. Sem evidência, sem conclusão. Sem contexto, sem sinal de atenção.

shadcn/ui

Código próprio, composto com rigor cívico

O DeOlho usa shadcn/ui como distribuição de código, não como caixa fechada. O componente base entra no repositório, recebe tokens do DeOlho e é composto em componentes cívicos revisáveis.

Open code

Agentes e humanos leem o componente real, revisam acessibilidade e ajustam comportamento sem wrapper frágil.

Monorepo consciente

`apps/web` consome `@workspace/ui`; shadcn base mora em `packages/ui`; features ficam no app.

Registry depois

Registro shadcn próprio só entra quando componentes cívicos tiverem Storybook, docs e uso real.

Anatomia

Todo componente cívico tem uma anatomia verificável

Componentes que mostram dado público precisam carregar estrutura de confiança, não apenas aparência. Esta anatomia deve orientar `EventoPublicoCard`, `ContratoCard`, `EntidadeCard` e páginas de entidade.

Card cívico modelo

Cabeçalho: tipo de informação + entidade afetada
Resumo neutro, sem acusação e sem inferência sem fonte
Badges: fonte, confiança, estado da coleta
Limitação explícita quando campo faltar ou fonte atrasar
Ações: ver evidência, acompanhar, citar, exportar
1. Identidade semântica

O usuário vê se é fato oficial, IA, sinal, notícia, opinião ou revisão manual antes da conclusão.

2. Proveniência

Fonte, data publicada, data coletada e link de evidência ficam no fluxo principal.

3. Confiança e limitação

Baixa confiança, dado incompleto, fonte atrasada e contradição têm linguagem própria.

4. Ação segura

Ação contextual não abre post, like ou comentário livre. Ela leva a evidência, correção ou exportação.

Precisa mostrar documento?

Use `EvidenciaLink` dentro de `PainelEvidencia`, com fonte e datas.

nunca apenas tooltip
Precisa explicar com IA?

Use `BlocoExplicacaoIA`, com fontes usadas e limitação. IA não entra como fato oficial.

label obrigatório
Precisa chamar atenção?

Use `SinalAtencaoCard`, com método, evidência, limitação e aviso de não irregularidade.

sem ranking moral

Estados

Nada entra só no estado feliz

Estados são parte do produto. Em dados públicos, ausência, atraso e contradição são informação.

Carregando

Skeleton com estrutura real, sem spinner solto em listas densas.

Vazio

Explica se não houve coleta, se o filtro não retornou ou se a fonte não informa.

Erro

Separa falha operacional de limitação da fonte ou da entidade.

Dado incompleto

Mostra campo ausente, fonte consultada e impacto na leitura.

Fonte atrasada

Mostra última coleta e não apresenta dado antigo como atual.

Contraditório

Mostra fontes em conflito e não escolhe vencedor sem regra documentada.

Maturidade

Componentes têm ciclo de vida público

Um design system confiável não trata todo componente como estável. Cada peça precisa dizer se é rascunho, candidata, estável ou depreciada.

draft

Rascunho

Existe para explorar. Pode mudar sem aviso. Só usa dados sintéticos e não bloqueia produto.

candidate

Candidato

Tem API proposta, Storybook mínimo, estados principais e documentação inicial.

stable

Estável

Tem uso em tela, acessibilidade revisada, exemplos completos e checklist de confiança cumprido.

deprecated

Depreciado

Tem substituto documentado, plano de remoção e bloqueio contra novo uso.

Promover para candidate

Exige props documentadas, dados sintéticos, loading, vazio, erro e fonte atrasada quando aplicável.

docs + Storybook
Promover para stable

Exige uso real em tela, revisão de acessibilidade, revisão de linguagem e teste de regressão.

PR checklist
Alterar componente stable

Exige changelog, justificativa, análise de impacto e screenshots desktop/mobile.

sem mudança silenciosa
Governança

Como humanos e agentes devem evoluir a UI

A governança evita componentes duplicados, decisões soltas e regressões de confiança. Toda mudança de UI precisa atravessar documentação, Storybook e revisão.

Ler contexto

`PROJECT.md`, `DESIGN_SYSTEM.md`, docs do Interface System e issue ativa.

Escolher componente existente

Procurar em `packages/ui`, `docs/components` e Storybook antes de criar algo novo.

Implementar com estados

Padrão, loading, vazio, erro, incompleto, fonte atrasada e baixa confiança quando fizer sentido.

Documentar e testar

Criar Storybook, atualizar docs e validar acessibilidade, linguagem e mobile.

Registrar handoff

Issue, arquivos alterados, verificações, pendências e próxima issue recomendada.
